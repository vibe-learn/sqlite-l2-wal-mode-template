        # sqlite — WAL-режим

        Homework-шаблон для урока **l2_wal_mode** (WAL-режим) на платформе Vibe Learn.

        ## Что делать

        На Go с modernc.org/sqlite напиши бенчмарк, сравнивающий rollback journal и WAL. Открой
одну файловую базу (не :memory:, т.к. WAL завязан на файл), включи нужный режим через
PRAGMA. Запусти один пишущий goroutine (UPDATE в цикле) и N читающих goroutine (SELECT
в цикле), замеряй p50/p99 латентности чтения в обоих режимах. Покажи, что под WAL read-p99
во время записи существенно ниже. Дополнительно: проверь, что `PRAGMA journal_mode`
действительно вернул "wal", и продемонстрируй ручной `PRAGMA wal_checkpoint(TRUNCATE)`.
Тесты проверяют, что режим включился и что измерения собираются корректно.

## Контекст (из transfer-задачи урока)

Read-heavy веб-приложение на одном сервере (Go, modernc.org/sqlite), база на дефолтном
rollback journal. Профиль нагрузки: ~95% запросов — чтение, ~5% — запись. Жалоба: под
пиком read-латентность скачет, p99 чтений деградирует именно в моменты записи; иногда
мелькает «database is locked». DevOps предлагает «переехать на PostgreSQL ради
конкурентности». Ты считаешь, что это преждевременно.

**Вопрос:** объясни:
(a) почему именно дефолтный rollback journal даёт всплески read-латентности при записи
    (что физически блокирует читателей);
(b) как WAL это чинит и почему для профиля 95% чтений это выигрыш №1;
(c) какой пресет PRAGMA включишь (journal_mode, synchronous, busy_timeout) и почему;
(d) честно: что WAL НЕ чинит (одиночный писатель остаётся), и при каком профиле нагрузки
    переезд на PostgreSQL действительно оправдан.

## Recap из урока

- `PRAGMA journal_mode=WAL` — **перформанс-выигрыш №1** для большинства приложений. Настройка persistent (записывается в заголовок файла).
- Главное свойство WAL: **читатели не блокируют писателя, писатель не блокирует читателей** (читают main+wal, писатель аппендит в -wal). В rollback journal писатель выгонял всех читателей.
- WAL **НЕ** делает SQLite многопишущей базой: писатель по-прежнему ОДИН. Это не MVCC, версий строк нет.
- Сайдкары: `-wal` (лог изменений) и `-shm` (shared memory индекс). **Checkpoint** сливает -wal в основной файл (авто ~1000 страниц).
- `synchronous=NORMAL` под WAL — безопасно и быстро. WAL **не работает по NFS/сети** (нужен mmap на локальном диске).

        ## Как работать

        1. Платформа Vibe Learn создаёт копию этого репо в твоём GitHub-аккаунте по клику «Начать домашку» на странице урока (через GitHub `/generate`, codecrafters-pattern).
        2. Склонируй копию локально, реализуй TODO в `main.go`, прогони тесты, запушь.
        3. CI (`.github/workflows/ci.yml`) запускает `go vet` + `go test ./...` на каждый push. Платформа слушает результат через webhook от GitHub Actions и обновляет статус домашки на странице урока.

        ## Локальное окружение

        - Go 1.22+
        - SQLite встроена — **никакого сервера и docker-compose**. БД это один файл (`DATABASE_PATH`) или `:memory:` в тестах. Драйвер `modernc.org/sqlite` — чистый Go, без CGO, так что CI собирается без компилятора C.

        ## Запуск

        ```bash
        # Прогнать тесты (бегут на :memory:-базе, ничего поднимать не нужно)
        go test ./...

        # Запустить main (создаёт схему, печатает marker; замени stub на реализацию)
        go run .

        # На файловой БД (нужно для WAL/concurrency-уроков):
        DATABASE_PATH=./app.db go run .
        ```
